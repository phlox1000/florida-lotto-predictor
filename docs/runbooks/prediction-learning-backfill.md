# Prediction Learning Metrics Backfill — Runbook

## Why this exists

`prediction_learning_metrics` (PLM) currently has 0 rows in production. The
live system populates it via `rebuildPredictionLearningMetricsFromEvents`,
which reads from `app_events` filtered to `prediction_accuracy_calculated`.
Historically, `app_events` has only ~324 such events, all clustered in a
short window. As a result, the brain's ensemble has no learned model
weights, and the live generator falls back to defaults.

This backfill produces accuracy events for the ~1,200 attributed historical
predictions in the `predictions` table by joining each prediction with
draws that occurred within 7 days after it (mirroring the live path's
behavior in `evaluatePredictionsAgainstDraw`). The synthetic events feed
the existing rebuild path, populating PLM in a way that's consistent with
how the live path would have populated it had events been emitted at the
time.

## Phase 1 vs Phase 2

This runbook covers **Phase 1**: emitting events for predictions that
*really happened*. The 1,200+ predictions in `predictions` are real, were
generated by real users, and are paired with real draw outcomes. The math
for each event's `match_ratio` is real. Only the timing of emission is
artificial — these events should have been emitted at the time of each
draw, but weren't, due to gaps in the original system.

**Phase 2** (counterfactual generation — running models against historical
draws to simulate predictions that never happened) is a separate decision
that has been deferred. Do not conflate the two. Phase 1 events are tagged
`source: "historical_backfill_phase1"`. Phase 2 events, if/when added,
will use a distinct source value to prevent silent miscoding.

## How to run the preview

From the repo root in PowerShell:

```powershell
# All games, all predictions, default 3 example payloads
pnpm preview:accuracy-backfill

# Single game
pnpm preview:accuracy-backfill -- --game-type=fantasy_5

# Single game, capped at 10 predictions
pnpm preview:accuracy-backfill -- --game-type=fantasy_5 --limit=10

# Date range
pnpm preview:accuracy-backfill -- --from=2026-04-23 --to=2026-05-04

# Verbose output (currently a no-op flag, reserved for future expansion)
pnpm preview:accuracy-backfill -- --verbose
```

The script reads from production `predictions` and `draw_results`. It
performs zero writes. It will reject `--commit`, `--write`, and
`--BACKFILL_SYNTHETIC_EVENTS` flags with a clear error.

## Important invariants the backfill preserves

- **`match_ratio` is main-numbers-only.** Special-number hits are tracked
  separately in `model_performance.specialHits` but do not contribute to
  `match_ratio`. The backfill mirrors this exactly via
  `scorePredictionAgainstDraw`.
- **`occurred_at` is the historical draw date, not insertion time.** The
  rebuild filters by `occurred_at`, so events with old `occurred_at`
  values may fall outside the live default 90-day rolling window. PR 3
  will run the rebuild explicitly after backfill writes.
- **Each prediction can match multiple draws.** The live path matches a
  prediction against every draw within 7 days after `prediction.createdAt`
  for the same gameType. The backfill mirrors this. For daily-draw games
  like fantasy_5, this means up to 7 events per prediction.
- **Only attributed predictions are backfilled.** Anonymous predictions
  (no `userId`) were never persisted to the `predictions` table to begin
  with, so they're not in the source population.

## Deliberate divergence from live behavior

The live `evaluatePredictionsAgainstDraw` caps prediction lookups at 200
rows per draw (`limit(200)`) for performance. The backfill does not apply
this cap. The cap is a runtime sanity limit, not a semantic part of the
rebuild logic; backfilling without it produces a *more complete* event
history than the live path would have, which is the desired outcome.

## Games with insufficient data (heads-up for PR 3 output)

Several games in the DB have shallow history or no factor_snapshot
coverage:

- **`cash4life`:** 18 predictions, 0 with factor_snapshot. Only model
  rollups will be produced for it.
- **`cash_pop`:** has 252 predictions in the table, but is not in
  `FLORIDA_GAMES`. Excluded entirely from backfill (matches the live
  path's behavior, which iterates `FLORIDA_GAMES` only via
  `learningMetrics.service.ts`).
- **`mega_millions`, `pick_2`, `pick_3`, `pick_4`, `pick_5`:** short
  history; factor metrics will be sparse but present.

## Expected output of Phase 1 (after PR 3 runs)

- `metricType="model"` rows for ~9 games × 18 models = up to ~162 rows.
  Each with sample counts in the 18–342 range, well above the 30-sample
  confidence threshold for most (game, model) cells.
- `metricType="factor"` rows for ~8 games × 12 factor names = up to
  ~96 rows. Sample counts will be lower (288 events distributed across
  12 factors × 8 games = ~3 per cell on average), so `weightedScore`
  will be uniformly discounted by the rebuild's `count/30` confidence
  factor. This is correct behavior, not a bug.

After Phase 1 ships, factor rollups will grow naturally as new attributed
predictions accumulate (the explainable scoring pipeline is now standard
in the live path).

## Running the write-capable backfill

After a clean preview, the write-capable script emits events for real:

```powershell
# Pilot one game first (dry-run validates expected event count)
pnpm backfill:accuracy-events -- --game-type=fantasy_5

# Pilot one game, commit
pnpm backfill:accuracy-events -- --game-type=fantasy_5 --commit=BACKFILL_PHASE1_EVENTS

# After validating PLM rows in DBeaver, run all games
pnpm backfill:accuracy-events -- --commit=BACKFILL_PHASE1_EVENTS
```

The exact phrase `--commit=BACKFILL_PHASE1_EVENTS` is required for any write. Any other value (including `--commit=true`, `--commit=yes`, or a missing value) aborts before touching the DB.

## Idempotency and re-runs

The backfill is safe to re-run.

Each event has a deterministic `id` of the form `backfill:accuracy:<gameType>:<predictionId>:<drawResultId>`. The primary key on `app_events.id` enforces uniqueness. The script's insert uses `ON DUPLICATE KEY UPDATE recorded_at = recorded_at` (no-op upsert), so collisions on re-run produce zero new rows and zero changes to existing rows.

Re-running picks up new (prediction, draw) pairs that became matchable since the previous run — for example, predictions made within the last 7 days whose draw windows have since filled in. The 54 unmatched fantasy_5 predictions noted in the preview will become processable as new fantasy_5 draws arrive over the following week. Re-running this script after that point captures them automatically.

## Rolling back

To remove all events from a specific backfill run:

```sql
DELETE FROM app_events
WHERE event_type = 'prediction_accuracy_calculated'
  AND JSON_UNQUOTE(JSON_EXTRACT(payload, '$.source')) = 'historical_backfill_phase1'
  AND JSON_UNQUOTE(JSON_EXTRACT(payload, '$.backfill_run_id')) = '<RUN_ID>';
```

To remove all Phase 1 backfill events (across all runs):

```sql
DELETE FROM app_events
WHERE event_type = 'prediction_accuracy_calculated'
  AND JSON_UNQUOTE(JSON_EXTRACT(payload, '$.source')) = 'historical_backfill_phase1';
```

After deletion, re-run `refreshPredictionLearningMetrics` to recompute PLM from the remaining events.

## Concurrency with the live path

If a real new draw arrives mid-backfill, the live `evaluatePredictionsAgainstDraw` will fire-and-forget call `rebuildPredictionLearningMetricsFromEvents`. PLM upsert is last-write-wins on the unique key, so:

- Transient state during a concurrent run may show a partial snapshot from the live rebuild
- The backfill's own per-game rebuild call is the last write, restoring correct state
- No data corruption is possible; only transient inconsistency

If observability of brain weights is critical during a backfill window, run during low-traffic periods. Otherwise, the architecture self-heals.
